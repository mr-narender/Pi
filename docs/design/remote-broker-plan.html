Remote Broker (#11) — pairing + client-held token

Watch / drive your active Pi chat from a phone. No Cloudflare Access. The QR is a one-time pairing bootstrap only — the server mints the session token at scan time, returns it once, and the phone keeps it client-side. No secret ever lives in a URL. 24h TTL + rotation. Pairing scheme — URL is a bootstrap, not the credential

QR / link: https://pi.fromlab.work/p#<pairing_code> └ single-use, ~90s TTL, PIN-gated, carries NOTHING secret On scan (phone JS reads the fragment, never sent as a GET param): POST /pair { pairing_code, device_pin } └─ server: validate (single-use + unexpired + PIN + rate-limit) → GENERATE session token = secrets.token_urlsafe(32) [256-bit, server-side, once] → bind { token → sid, device }, exp = now + 24h → mark pairing_code CONSUMED → return { token } ONCE in the JSON body phone: store token in IndexedDB (client-side only) → open wss /view with the token in the WS auth frame (never in a URL)
  • The session token is minted at scan time, returned once, never re-issued — the standard device-pairing / code-exchange pattern.
  • It is held only on the client and sent only in the WS auth frame — never in a URL, query, path, access log, or Referer.
  • The pairing code is single-use: a screenshotted QR is dead the moment it's scanned; if never scanned it dies in ~90s.
  • Page hardening: strict Content-Security-Policy (no third-party scripts), Referrer-Policy: no-referrer, HSTS — protects the client-stored token.

24h lifecycle & rotation

t=0 Start Remote Session → session created; QR (pairing code) + PIN shown in VS Code scan Phone pairs → server mints token (exp = now+24h) → phone stores it client-side t<24h Phone reconnects with its stored token (no re-scan needed within the window) Refresh Rotate → server issues a NEW token to paired devices; OLD token invalidated immediately t=24h Auto-expire → token purged, room closed, phone disconnected → re-pair or refresh Stop Manual kill → same as expire, instantly; all device tokens revoked

The token lives on the client for up to 24h; rotation is the only way to extend — no silent renewal.

Security model (Access removed)

Layer Role
1 · TLS (wss) + Cloudflare proxy Encryption + DDoS shield + WAF rate-limit rules at the edge.
2 · One-time pairing code + device PIN Bootstrap only. Single-use, ~90s, PIN-gated, rate-limited. Redeeming it is the sole way to obtain a token.
3 · Server-minted session token (256-bit) Generated at pairing, returned once, stored client-side, sent only in the WS auth frame. 24h TTL, rotatable, session-scoped. Never in a URL.
4 · Roles + driver-lock Host token is private (not the pairing flow). Viewer = read-only; one driver; tool approvals still prompt; instant revoke/kill.

Threat analysis (public endpoint)

Attack Mitigation
No secret in the URL at all The token is minted server-side and stored client-side — it never touches a URL/query/path/log/Referer. Eliminates the whole capability-URL leak class.
Brute-force the token 256-bit random, server-side only → infeasible.
QR forwarded / screenshotted Pairing code is single-use (dead once scanned) + ~90s TTL + device PIN to redeem. A stale screenshot is worthless.
Guess a pairing code High-entropy code + single-use + ~90s + per-IP rate-limit/lockout + edge WAF.
Stolen client token (XSS) Strict CSP + no third-party scripts on the PWA; token in IndexedDB; 24h expiry; host can drop devices.
Cross-session hijack Token binds to its sid, validated per room. A token for chat A is rejected by chat B.
Impersonate the host Host uses a separate private token, never the pairing flow. Viewers can only view/drive.
Scan / flood the domain No token → handshake rejected instantly. Cloudflare absorbs floods; hard cap on active sessions + connections/session.

Because the credential is minted at scan and kept client-side, a leaked/forwarded QR is a single-use, short-lived, PIN-gated bootstrap — not a standing backdoor. This is the standard, safe pattern.
