Remote Broker (#11) — architecture

Watch / drive your active Pi chat from a phone. Broker runs on slr-backend (FastAPI), exposed at pi.fromlab.work. Plan — not yet built.

Your questions, answered

1 · Is this a WS connection from VS Code to an external URL?

Yes. The extension is a client that dials OUT to wss://pi.fromlab.work/host. Nothing listens on the Mac; no inbound; works on any network. The phone is also a client (/view). The broker on slr-backend is the hub that relays between them. It never talks to Pi directly.

2 · How is security implemented?

Four independent layers — see “Security model” below. TLS everywhere, Cloudflare Access at the edge, a host token that only the extension holds, and short-lived session-scoped viewer tokens with a driver-lock.

3 · One-time code + URL — brute-force / cross-session hijack?

The URL is fixed and public; it grants nothing on its own. Each session gets a 128-bit random id (unguessable). The pairing code is single-use, 90-second TTL, bound to one session, rate-limited, and (with Access on) only your SSO identity can even reach the endpoint. Tokens are session-scoped, so a token for one chat can never touch another. See “Anti-hijack” below.

4 · Runs on slr-backend — pattern reused?

Yes. slr-backend is FastAPI and already ships this exact shape in its heartbeat app: WebSocket sessions authenticated via a session_token query param, an in-memory _rooms dict (room = session) with a lock, secrets tokens, and verify_assertion for device auth. The broker is a new small app that copies this.

Session + pairing flow

1 VS Code: Pi: Start Remote Session 2 Extension → POST /session (host token) → broker mints session_id = secrets.token_urlsafe(24) [192-bit] 3 Extension opens wss /host?session=<id> and streams WebviewSnapshots up 4 Extension shows a QR + 6-digit code (popover). Broker stored: code → session_id, single-use, TTL 90s 5 Phone: open pi.fromlab.work → Cloudflare Access SSO (your email) → scan QR / type code 6 Phone → POST /pair {code} → broker verifies (single-use, unexpired, rate-limited) → issues viewer JWT: {sid, role:spectator, exp:+30m}, signed 7 Phone opens wss /view?session=<id> with the viewer token → joins the room → live chat 8 Promote to driver: explicit tap → broker grants driver-lock (only if free). Desktop can reclaim anytime. 9 Pi: Stop Remote Session → room closed, all tokens invalidated, host socket dropped.

Security model (four layers)

Layer What it stops
1 · TLS (wss) end-to-end Eavesdropping / MITM. Cloudflare terminates TLS; tunnel to origin is encrypted.
2 · Cloudflare Access (edge SSO) Anyone who isn't you — the endpoint is invisible without a valid slrinc SSO session. Blocks the internet before a request ever reaches the broker.
3 · Host token Only the extension (holding the vault token) can publish snapshots for a session. A viewer can never impersonate the host.
4 · Viewer token + roles Session-scoped, signed, expiring. Spectator = read-only. Driver-lock = one writer. Tool approvals still prompt inline. Host can revoke; stop = invalidate all.

Anti-hijack / brute-force analysis

Attack Why it fails
Guess the URL URL grants nothing. No session, no token → rejected at handshake.
Guess a session id 192-bit random (secrets.token_urlsafe(24)). Not brute-forceable.
Brute-force the 6-digit code Single-use + 90s TTL + rate limit (e.g. 5 tries → lock the code). ≤ ~1e-5 chance in the window. Prefer the QR (long token, no typing) — code is the fallback.
Cross-session hijack Every token embeds its sid and is validated against the room. A token for chat A is rejected by chat B. Rooms are isolated by the random id.
Replay a captured token Short exp (30m), bound to device on first use, revocable; session end invalidates all. Access cookie also required.
Reach the broker at all Cloudflare Access (layer 2) already blocked non-you identities before any of the above.

Net: an attacker needs a valid slrinc SSO session and a 192-bit session id and an unexpired single-use pairing code — practically impossible.
