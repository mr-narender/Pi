Remote Broker (#11) — public capability-URL architecture

Watch / drive your active Pi chat from a phone via a public, unguessable link. No Cloudflare Access. Security = a 256-bit capability URL, 24h TTL + rotation, and an optional device PIN. Separate slr-inc app on pi.fromlab.work. Plan — not built.

Confirmed requirements

Requirement Design
Public — no CF Access gate Cloudflare proxy ON (DDoS + WAF), Access OFF. Anyone can reach the host; only a valid capability token gets a session.
Very safe URL (session scheme) 256-bit secret in the URL fragment (never logged / never in Referer). https://pi.fromlab.work/j#<sid>_<secret>
Valid 24h, then rotate Token TTL = 24h. “Refresh link” mints a new secret, instantly kills the old, +24h. Auto-expire + disconnect at 24h if not refreshed.

The URL scheme (why the fragment matters)

https://pi.fromlab.work/j#<sid>_<secret> │ │ public 8-char routing id ┘ └ 256-bit secret (secrets.token_urlsafe(32))
  • Everything secret lives after the # → the fragment is never sent to the server, so it can't appear in Cloudflare/nginx/app access logs or a Referer header.
  • The phone's JS reads location.hash, then sends the secret only inside the WS auth frame (which we never log).
  • Referrer-Policy: no-referrer, HSTS, no third-party scripts on the page → no cross-origin leak of the link.
  • sid is public and used only for routing + rate-limit keying; it reveals nothing without the secret.

24h lifecycle & rotation

t=0 Start Remote Session → mint {sid, secret}, exp = now+24h → link shown (QR + PIN) t<24h Link works. Phone joins with #secret (+ PIN on first device). Refresh Rotate: new secret, exp = now+24h, OLD secret invalidated immediately (old link dies). t=24h Auto-expire: room closed, phone disconnected, token purged. Must Start/Refresh again. Stop Manual kill: same as expire, instantly.

One active secret per session at a time. Rotation is the only way to extend; there is no silent renewal.

Security model (Access removed — 3 layers + PIN)

Layer Role
1 · TLS (wss) + Cloudflare proxy Encryption + DDoS shield + WAF rate-limit rules at the edge.
2 · Capability token (256-bit, fragment) The credential. Unguessable, leak-hardened, 24h TTL, rotatable, session-scoped.
3 · Roles + driver-lock Host token is private (not the public link). Viewer = read-only; one driver; tool approvals still prompt; instant revoke/kill.
+ Device PIN (recommended) 6-digit PIN shown in VS Code, required on first join per device → issues a device sub-token. A leaked link alone cannot join.

Threat analysis (public endpoint)

Attack Mitigation
Brute-force the secret 256-bit random → infeasible. Plus per-sid join rate-limit + lockout, and edge WAF rate rules.
Scan / hit the domain No token → handshake rejected instantly. Cloudflare absorbs floods; hard cap on active sessions + connections/session.
Link leaks (log / Referer) Secret is in the fragment → not in any server log or Referer. no-referrer + HSTS.
Link forwarded / screenshotted main residual risk Device PIN on first join blocks it; 24h expiry + rotation limits the window; host sees connected devices and can revoke.
Cross-session hijack Token embeds sid, validated per room. A link for chat A is rejected by chat B.
Impersonate the host Host uses a separate private token, never the public link. Viewers can only view/drive, never publish as host.

Without Access, a shared link is the whole credential — so the device PIN + 24h rotation are what keep an accidentally-forwarded link from becoming a standing backdoor. Strongly recommended to keep the PIN on.
