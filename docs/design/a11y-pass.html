Accessibility Pass — what changes

Pi chat (VS Code webview). Goal: usable with keyboard + screen reader, WCAG 2.1 AA.

1 · Streaming announcements (ARIA live)

The transcript re-renders many times a second while Pi types. A screen reader currently tries to read every partial update — a flood of noise.

before#messages has aria-live="polite" → SR announces every token chunk (unusable).
after Transcript set to aria-live="off". A dedicated hidden status region announces concise milestones only:
  • “Pi is working…” when a turn starts
  • “Pi responded. <first ~220 chars of the answer>” when the turn settles
  • “Error: …” on failure
aria-live="polite" region → “Pi responded. Here's how photosynthesis works…”

WCAG 4.1.3 Status Messages · 1.4.13 · aria-live=polite (non-interrupting)

2 · Labels on icon-only controls

Every icon button gets an aria-label (not just a hover title).

← Tab through these; each announces its purpose. WCAG 4.1.2 / 1.1.1

3 · Focus management & visible focus

Focus rings
Every interactive element shows a visible :focus-visible outline (2px teal). Never removed. WCAG 2.4.7
Skip link
“Skip to composer” already first in tab order — kept.
Message authorship
Each message exposes its author to SR (aria-label="You said" / "Pi said") so the reader knows who is speaking.
Menus
Popovers (More, Settings) are keyboard-openable and Esc-closable; focus returns to the trigger.

4 · Reduced motion

prefers-reduced-motion: reduce now also disables the typewriter smoothing (shows text immediately) in addition to the spinner/CSS animations already covered. WCAG 2.3.3

Summary of code touch-points

  • render.ts — hidden live region; per-message author labels; aria-labels on chips/buttons
  • chat.ts — announce working/settled/error into the live region; typewriter honours reduced-motion
  • chat.css — :focus-visible outlines; .visually-hidden live region
